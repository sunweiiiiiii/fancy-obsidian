# SZJ 面部医疗超声 AI 项目：MTK 8300 边缘端部署架构评估白皮书

## 0. 项目背景与工程约束
* **核心业务**：面部美学 14 分区高频超声影像的实时识别与分割（涵盖筋膜层、脂肪层、肌肉层等连续解剖结构）。
* **数据通量**：单病人高频连续扫描，约 6,000 帧/病例。
* **目标硬件**：基于 MediaTek (MTK) Dimensity 8300 (APU 780) 的边缘计算节点。
* **工程目标**：实现可控的延迟抖动（Latency Jitter）、INT8 工业级精度，并最终与 SQLite 异步持久化架构及 Tauri 前端集成。

---

## 1. 核心架构共识：从“纯 NPU”转向“异构计算”

在早期的架构构想中，常误将“100% 算子映射至 APU、零 CPU Fallback”设定为硬性 KPI。针对 MTK 8300 这一偏向手机端生成式 AI 的 SoC，结合高频医疗流媒体的特性，现将核心架构修正为**异构协同计算（Heterogeneous Computing）**。

### 1.1 硬件算力分配策略
* **NPU/APU 职责**：仅负责主干网络（Backbone）及解码器（Decoder）的密集矩阵乘加（Conv/GEMM）运算。必须保持**静态 Shape** 以最大化寄存器调度效率。
* **CPU/NEON 职责**：负责所有动态逻辑、小 Tensor 操作、图像解码以及重度后处理（如 YOLO 架构下的 NMS、阈值过滤、区域生长等）。
* **防御性设计**：医疗实时系统最大的敌人不是“平均帧率低”，而是“延迟抖动”。允许受控的 CPU 参与，比强行把动态逻辑塞进 APU 导致内存带宽阻塞更为稳健。

---

## 2. 模型选型评估：YOLO vs. 纯语义分割

针对面部超声中**动态范围窄、边界模糊、散斑噪声（Speckle Noise）强**的物理特性，重新评估模型底座。

### 2.1 YOLO 架构的潜在风险 (Object-Centric)
* **认知局限**：YOLO 的底层哲学是“目标检测”。它强依赖于寻找独立物体的 Bounding Box。
* **超声不适配**：面部 14 分区中的 SMAS 层或肌肉间隙是“连续解剖结构（Continuous Anatomical Structure）”，而非独立物体。YOLO 在无明确边界的灰度图上极易产生检测框抖动，导致 Mask 边缘频繁闪烁。
* **后处理沉重**：NMS（非极大值抑制）对 CPU 造成的波峰负载，在 6,000 帧的连续处理中极易引发热节流（Thermal Throttling）。

### 2.2 推荐架构：像素级语义分割 (Pixel-Centric)
* **Attention-UNet / BiSeNet**：纯卷积架构。对 MTK NPU 极其友好，算子融合率极高，且跳跃连接（Skip-connection）能有效保留极细的筋膜边界。
* **SegFormer (B0/B1)**：轻量级 Transformer 编码器 + 极简解码器。对低对比度边界的全局纹理捕捉能力强，且在固定输入尺寸配合 INT8 QDQ 时，表现出极高的工程收敛性。

---

## 3. INT8 量化全链路治理：预处理漂移 (Preprocessing Drift)

INT8 量化不是单点的模型格式转换，而是贯穿训练、编译到 C++ 运行时的系统级工程。超声影像对灰度极其敏感，最大的量化灾难来源于**预处理漂移**。

### 3.1 漂移的来源与危害
* **插值漂移 (Resize Drift)**：Python 端的 `cv2.resize` 与 C++ 端底层 SIMD 实现的微小差异。
* **舍入漂移 (Rounding Drift)**：浮点数标准化转换为 INT8 时，Python 的“银行家舍入”与 C++ 标准的微小差异，可导致筋膜层判定错误。

### 3.2 黄金预处理基线 (Golden Preprocessing Pipeline)
必须建立独立的**校准数据准备工具**：
1. **统一数据源**：从原始采集流中抽取 200-500 张涵盖 14 分区多角度的代表帧。
2. **强制像素对齐**：使用与最终 C++ 推理代码**绝对一致**的 Resize、均值/方差标准化逻辑。
3. **输出校验**：确保校准工具输出的数据矩阵与 C++ 运行时预处理模块输出的数据，其 MD5 值或位级（Bit-wise）比对完全一致。

## 4. 阶段性交付路线图 (Phased Roadmap)

针对复杂的系统栈（Tauri + Rust + C++ + SQLite），必须进行解耦和降级验证，防止早期深陷架构泥潭。

### Phase 1：纯净推理链路验证（耗时验证与抖动排查）
* **目标**：验证 MTK 8300 APU 真实的单帧延迟、发热降频拐点与 mIoU 精度。
* **链路**：`Raw Frame` -> `Golden Preprocess (C++)` -> `INT8 .dla 推理` -> `Raw Mask`。
* **动作**：暂缓所有 UI 和数据库开发。通过独立 C++ CLI 工具，压测 6,000 帧数据的吞吐能力与内存泄露情况。

### Phase 2：数据流稳定与持久化
* **目标**：解决高频数据的 IO 压力。
* **链路**：引入异步队列与单文件数据库设计。
* **动作**：实现基于 C++ 的 SQLite 异步写入。将 APU 推理出的 Mask 进行 RLE (Run-Length Encoding) 压缩，配合帧时间戳（Frame Index）存入数据库，解决 mmap 压力和 SSD 写放大问题。

### Phase 3：应用层集成
* **目标**：完成医疗端产品的可用性包装。
* **动作**：引入 Tauri (Rust) 框架，实现基于状态机的扫描引导（如：自动识别横纵切面转换），完成前端实时渲染流。

---

## 5. 总结与第三方评估诉求

MTK 8300 可以胜任本项目的推理任务，但需严格遵守“静态输入、轻量分割模型、精细化异构协同”的原则。对于后续的技术落地，现向第三方评估团队提出以下论证诉求：

1. **针对超声伪影的选型**：在 SegFormer 与 Attention-UNet 之间，结合 MTK NeuroPilot 工具链的支持度，寻求更具确定性的算子通过率评估。
2. **零拷贝内存管理**：在 Android/Linux 底层，针对 8300 平台的 Ion Memory 或 DMABUF 机制，评估其在 6,000 帧连续输入下的稳定性。
3. **工业级芯片备选方案评估**：若 8300 在 Phase 1 压测中出现不可控的热降频，是否需要立刻启动基于 MediaTek Genio 系列（偏向工控与边缘计算平台）的备选迁移计划。
